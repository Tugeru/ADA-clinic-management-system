Create a high-fidelity, responsive clickable prototype in Figma for the ADA system (Patient Records + Visit Logging core flow).

PROJECT
Name: ADA – Patient Records & Visit Logging
User role: Clinic In-Charge (single role)
Patient types supported: Student, Teacher, Non-Teaching Personnel (NTP)
Style: modern clinical UI, calm/minimal, clear hierarchy, consistent components

DESIGN SYSTEM (apply consistently)
- Font: Inter
- Colors:
  - Primary: #2A9D8F
  - Secondary: #3A86FF
  - Background: #F4F6F8
  - Card: #FFFFFF
  - Success: #2ECC71
  - Warning: #F4A261
  - Danger: #E63946
- Spacing: 8px system (8/16/24/32)
- Radius: 8px
- Shadows: subtle card shadow (light elevation)
- Components must be Auto Layout friendly (cards, tables, form sections, buttons)
- Use reusable components/variants for: Header, Sidebar, Buttons, Badges, Inputs, Tables, Tabs

UNIFORM GLOBAL SHELL (MUST APPEAR ON ALL PAGES)
1) UNIFORM HEADER (same position, height, and structure across all screens)
- Height ~64px, full width
- Left: Page title (changes per screen)
- Middle/right: Global search input (“Search patients, visits, medicines…”)
- Right: User menu “Clinic In-Charge” (avatar + dropdown chevron)
- Header is fixed/sticky for desktop and mobile variants

2) UNIFORM SIDEBAR NAVIGATION (same items and layout across all screens)
- Desktop/Tablet: fixed left sidebar width ~260px
  - Logo “ADA”
  - Nav items: Dashboard, Patients, Visits, Inventory (disabled), Analytics (disabled), Archive (disabled), Settings (disabled)
  - Highlight active nav item
- Mobile: sidebar becomes a hamburger menu in the header
  - Hamburger icon opens an overlay drawer containing the same nav items
  - Drawer includes close icon and active item highlight

RESPONSIVE REQUIREMENTS (MUST INCLUDE)
Create responsive variants for each screen:
- Desktop: 1440x900 (primary)
- Mobile: 390x844 (iPhone 14 size)
Responsiveness rules:
- Desktop: sidebar visible; multi-column layouts allowed
- Mobile: sidebar hidden behind hamburger; content is single-column stacked; tables become:
  - either stacked row cards OR horizontal scroll with sticky headers (choose one and apply consistently)
- Use Auto Layout + constraints to ensure components adapt between Desktop and Mobile variants

SCREENS TO GENERATE (7) + RESPONSIVE VARIANTS (DESKTOP + MOBILE)
1) Dashboard
2) Patient List
3) Add/Edit Patient (one screen with an “Add” and “Edit” state)
4) Patient Profile (tabs: Information / Visit History)
5) Visit List
6) New Visit (medicine table + conditional guardian section)
7) Visit Details

SCREEN DETAILS

1) DASHBOARD
- KPI row (4 cards): Today’s Visits, Active Patients, Low Stock Count (placeholder), Total Medicines (placeholder)
- Quick Actions card: New Visit (primary), Add Patient (secondary)
- Today’s Visits preview table (3 rows) with “View all” link to Visit List

Mobile:
- KPI cards stack 2 per row or 1 per row (choose best readable)
- Quick Actions becomes full-width buttons
- Visits preview becomes stacked list or horizontal scroll table

2) PATIENT LIST
Top controls card:
- Search field: “Search by name or ID number…”
- Filters: Patient Type (All/Student/Teacher/NTP), Archived Status (Active/All)
Data table card:
- Columns (desktop): ID Number, Full Name, Type badge, Context (Grade/Section or Department/Unit), Status badge, Actions (View/Edit/Archive)
Empty state layer (hidden): “No patients found” + “Add New Patient”

Mobile:
- Table becomes stacked patient cards showing: Name, Type badge, ID, Context, Status, and action buttons

3) ADD/EDIT PATIENT (ONE SCREEN, TWO STATES)
- Single-column form card with sections:
  A) Patient Type (segmented control)
  B) Identification & Basic:
     ID Number (required), ID Type (dropdown), Full Name (required), Gender (required), DOB (optional)
     Conditional:
       Student: Grade Level (required), Section (required)
       Teacher/NTP: Department/Unit (required), Position/Role (optional)
  C) Emergency Contact / Notified Person:
     Contact Name (required), Relationship (required), Contact Number (required)
  D) Medical Notes (optional):
     Allergies, Known Conditions, Remarks
Footer actions: Save (primary), Cancel (ghost)
Include hidden validation layer (required-field errors)

Mobile:
- Same form, stacked, full-width inputs, sticky bottom action bar (Save/Cancel)

4) PATIENT PROFILE
Header card:
- Name, Type badge, ID Number, Context line (Grade/Section OR Department/Unit), Status badge
- Actions: Edit (secondary), Start New Visit (primary), Archive (icon placeholder)
Tabs:
- Information tab: two-column on desktop (Identification vs Contact), notes card below
- Visit History tab: table (Date, Complaint, Disposition, Time In/Out, View)
Empty state layer (hidden): “No visits recorded yet” + “Start First Visit”

Mobile:
- Profile header stacks; actions become full-width buttons
- Information tab becomes single column
- Visit history becomes stacked visit cards

5) VISIT LIST
Top controls card:
- Search by patient name
- Filters: Date Range (placeholder), Patient Type, Disposition (Returned/Sent Home)
Table:
- Columns: Date, Time In, Patient Name, Type badge, Complaint (truncate), Disposition badge, Actions
Disposition rules:
- Returned label dynamic:
  Student -> “Returned to Class”
  Teacher/NTP -> “Returned to Work”
- Sent Home badge uses Warning #F4A261
CTA: New Visit (primary)

Mobile:
- Visit rows become stacked cards with key fields + View/Edit buttons

6) NEW VISIT
Single-column form card with sections:
A) Patient Selection: dropdown + selected patient header strip (Name, Type badge, ID, Context)
B) Visit Details:
   Date (default), Time In (auto), Time Out, Chief Complaint (required), Assessment/Intervention, Remarks
C) Medicine Used:
   table rows with Medicine dropdown + Quantity + Remove; “+ Add Medicine”
D) Disposition:
   Radio: Returned (dynamic label) / Sent Home
   Conditional:
     If Sent Home AND Patient Type = Student: Released-to Parent/Guardian sub-card (required fields)
     If Sent Home AND Patient Type = Teacher/NTP: Notified Person sub-card (optional fields)
Footer actions: Save (secondary), Save & Finalize (primary), Cancel (ghost)

Validation layers (hidden):
- Missing complaint: red helper text + optional top danger banner
- Insufficient stock: row-level error + disable Save & Finalize

Mobile:
- All sections stacked
- Medicine used section becomes stacked rows (each medicine row as a mini-card) OR horizontal scroll table (choose one consistently)
- Sticky bottom action bar (Save/Finalize/Cancel)

7) VISIT DETAILS
Header card:
- Patient identity + type badge + context + disposition badge
- Actions: Edit, Archive, Print/Export (placeholder)
Body cards:
- Visit Summary
- Medicines Used table (or “No medicine dispensed”)
- Disposition Details card (guardian/notified info if applicable)

Mobile:
- All cards stack
- Medicines list becomes stacked items

PROTOTYPE INTERACTIONS (MUST BE CLICKABLE ON DESKTOP + MOBILE)
- Sidebar nav (desktop) / hamburger drawer nav (mobile):
  - Dashboard -> Dashboard
  - Patients -> Patient List
  - Visits -> Visit List
- Dashboard:
  - New Visit -> New Visit
  - Add Patient -> Add/Edit Patient (Add state)
  - View all -> Visit List
- Patient List:
  - Add Patient -> Add/Edit Patient (Add state)
  - View -> Patient Profile (use a Student row)
  - Edit -> Add/Edit Patient (Edit state, prefilled)
- Patient Profile:
  - Edit -> Add/Edit Patient (Edit state)
  - Start New Visit -> New Visit (patient preselected)
- Visit List:
  - New Visit -> New Visit
  - View -> Visit Details
  - Edit -> New Visit reused as Edit Visit (prefilled)
- New Visit:
  - Save & Finalize -> Visit Details
  - Error-state path: Save & Finalize with Missing Complaint layer visible -> stays on New Visit (shows error)
  - Error-state path: Save & Finalize with Insufficient Stock layer visible -> stays on New Visit (shows error)
